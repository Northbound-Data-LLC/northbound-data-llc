dbt Best Practices

5 Signs Your dbt Project Needs an Intervention

Most dbt projects start with good intentions. A few staging models, some clean transformations, maybe even a test or two. Then reality happens. Here are the warning signs that yours has crossed the line from "could use some cleanup" to "needs serious help."

1. Only One Person Understands How It Works

This is the most dangerous sign, and it's incredibly common. One person—usually whoever built the initial project—holds all the knowledge of how things work. They know why that one model has a weird join. They know which tests are actually important and which are noise. They know which models are used and which are abandoned experiments.

When that person goes on vacation, everyone holds their breath. When they leave the company, panic ensues.

The fix isn't just documentation (though that helps). It's building a project that's understandable by design: clear naming conventions, consistent patterns, and logic that's straightforward enough that a new team member can trace it without a guide.

2. dbt Runs Take Hours

If your full dbt run takes more than 30 minutes, something is wrong. If it takes hours, something is very wrong.

Long run times usually indicate one or more of these issues:

  • Excessive full table scans — Models that could be incremental aren't
  • Inefficient model dependencies — The DAG has unnecessary dependencies causing serialization
  • Duplicate work — The same transformations happening in multiple places
  • Overly complex models — Logic that should be split across layers is all crammed into one massive SQL file

Beyond the obvious cost implications (compute isn't free), long run times slow down development. When developers can't iterate quickly, they stop iterating at all.

3. Tests Are Failing and Nobody Cares

Test failures should cause concern. If your Slack channel has test failure alerts that everyone ignores, you don't have a testing problem—you have a cultural problem that started as a technical one.

This usually happens because:

  • Tests were added without understanding what they actually validate
  • Tests are too sensitive (alerting on acceptable data variance)
  • There are so many failures that they've become noise
  • Nobody knows how to fix the underlying issues

The solution involves triaging: which tests actually matter? What should their thresholds be? And critically—fixing the issues that tests surface, not just the tests themselves. A data quality and testing overhaul can systematically address this.

4. You Have Models That Nobody Uses

Every dbt project accumulates cruft. An analyst builds a model for a one-off analysis, then moves on. Someone experiments with a new approach and abandons it. A stakeholder requests something, priorities change, and the model is never connected to anything.

A few orphan models are normal. Dozens are a problem. They slow down runs, confuse new team members, and make the project harder to navigate.

The dangerous version of this is models that look important but aren't—they have downstream dependencies that themselves are unused. Tracing true usage requires more than just looking at the DAG; it requires understanding what actually powers decisions.

5. Every Change Feels Risky

This is the meta-sign that encompasses all the others. When your team hesitates to modify models because "something might break," your project has lost its purpose. dbt exists to make transformation manageable, not terrifying.

The fear usually comes from:

  • Lack of test coverage—no confidence that changes are safe
  • Unclear dependencies—not knowing what downstream will be affected
  • Missing documentation—not understanding what the code is supposed to do
  • Past trauma—previous changes caused incidents

A healthy dbt project should feel safe to change. Not recklessly—you should still test, review, and deploy carefully—but the mechanics of making changes should be straightforward and predictable.

What To Do About It

If you recognized your project in these signs, the path forward depends on severity.

If it's one or two issues: You might be able to address them incrementally. Block out time for tech debt, establish new patterns, and migrate towards them over time.

If it's all five: Incremental improvement is likely too slow. The project has accumulated so much entropy that you're better off with a structured rebuild. That doesn't mean throwing everything away—it means systematically redesigning the architecture while migrating the valuable logic.

The worst approach? Doing nothing and hoping it gets better. It won't. Technical debt compounds. The longer you wait, the harder the fix.
